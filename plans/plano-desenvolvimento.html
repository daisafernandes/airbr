Fase 0
Saneamento e Fundação
1–2 dias Agentes A + C + D (paralelo)

Corrigir inconsistências do scaffold atual antes de qualquer desenvolvimento novo. Sem isso, os próximos agentes vão bater em erros silenciosos.

Agente A — Backend
Corrigir Backend
  • Adicionar bcryptjs e @types/bcryptjs ao package.json do backend
  • Implementar GET /api/v1/users/:id (frontend já chama, backend não tem)
  • Adicionar jsonwebtoken ao package.json (JWT_SECRET já é validado no env)
  • Implementar middleware de autenticação JWT (authMiddleware.ts)
  • Criar rota POST /api/v1/auth/login com JWT response
  • Validar que DATABASE_URL não quebra o boot quando sem DB (modo dev)
Agente C — Frontend
Corrigir Frontend
  • Instalar tailwindcss, postcss, autoprefixer e criar configs
  • Criar rota /login + página LoginPage.tsx (axios redireciona para cá no 401)
  • Criar página /register + RegisterPage.tsx usando useCreateUser
  • Conectar AuthContext.signIn ao endpoint real POST /auth/login
  • Adicionar serviço authService.ts com login/logout
Agente D — DevOps
Configuração Base
  • Criar docker-compose.yml na raiz com PostgreSQL 16 + PostGIS + Redis
  • Adicionar jest.config.ts no backend (Jest + ts-jest)
  • Adicionar vitest.config.ts no frontend
  • Configurar GitHub Actions básico: lint + type-check em cada PR
  • Criar .env de exemplo completo com comentários por variável
bcryptjs instalado POST /auth/login /login page Tailwind funcionando Docker Compose (Postgres + Redis) CI/CD básico
Fase 1
Persistência e Primeiros Dados Reais
3–5 dias Agentes A + B (paralelo)

Conectar ao banco real e trazer os primeiros dados externos. O Agente A prepara o banco enquanto o Agente B constrói os coletores das APIs gratuitas mais simples.

Agente A — Backend & Infra
Banco de Dados e ORM
  • Instalar Prisma (ORM) com suporte a PostGIS via extensão raw
  • Criar schema Prisma: City (id, name, state, lat, lng, source), AqiReading (cityId, aqi, pm25, pm10, o3, no2, co, uv, pollen, timestamp, source), FireFocus (lat, lng, intensity, satellite, detectedAt, biome), User (completo)
  • Criar migrations e rodar no Docker Compose local
  • Trocar InMemoryUserRepository pelo PrismaUserRepository
  • Instalar ioredis e criar RedisCache service com TTL por entidade
  • Criar CacheService com métodos get/set/invalidate e fallback gracioso (sem redis = sem cache)
Agente B — Data Pipeline
Primeiros Coletores (sem chave de API)
  • Criar estrutura de coletores: src/jobs/collectors/ com interface ICollector
  • Implementar OpenWeatherMapCollector: AQI por coordenada, 1k req/dia grátis
  • Implementar AQICNCollector: +500 estações BR, token gratuito
  • Implementar INPEFiresCollector: focos de queimada CSV, polling 3h
  • Implementar OpenMeteoCollector: clima + UV + AQI, sem chave de API
  • Criar Normalizer: converte respostas díspares para o schema unificado {city, aqi, pm25, pm10, uv, pollen, fires, lat, lng, timestamp, source}
  • Criar lista inicial de 50 cidades brasileiras monitoradas em data/cities.json
⚠️ O Agente B depende do schema do Prisma para salvar. Alinhar os tipos do Normalizer com o schema antes de começar a escrever no banco. Recomendado: Agente A entrega migrations antes do Agente B iniciar a escrita.
Prisma + PostgreSQL Redis CacheService OpenWeatherMap collector AQICN collector INPE fires collector Open-Meteo collector Normalizer unificado
Fase 2
Scheduler + API Core
3–4 dias Agentes A + B (paralelo)

Automatizar a ingestão de dados e expor os primeiros endpoints REST para o frontend consumir. É o ponto central de geração de valor do produto.

Agente B — Data Pipeline
Scheduler de Jobs
  • Instalar node-cron e criar JobScheduler com registro de coletores
  • Configurar cron: AQI a cada 1h, Queimadas INPE a cada 3h, UV/clima a cada 1h, Limpeza de cache a cada 24h
  • Criar JobLog no DB: registrar cada execução (status, duração, registros inseridos, erros)
  • Adicionar rate-limiting interno: respeitar limites das APIs gratuitas (ex: 1k/dia OWM)
  • Implementar retry com exponential backoff para falhas de rede
  • Adicionar endpoint GET /api/v1/admin/jobs protegido com JWT para status dos jobs
Agente A — Backend
Endpoints REST Core
  • GET /api/v1/cities — lista com AQI atual, lat/lng, fonte. Cache Redis 15min
  • GET /api/v1/cities/:id — dados completos: AQI, PM2.5, PM10, O₃, NO₂, CO, UV, pólen, alertas de saúde, fonte
  • GET /api/v1/cities/:id/history — histórico por período (?period=7d|30d|1y)
  • GET /api/v1/fires — focos ativos, filtros por estado e bioma
  • GET /api/v1/ranking — top 10 mais e menos poluídas, filtros por região/estado
  • GET /api/v1/search?q= — autocomplete por nome de cidade com estado e AQI atual
  • GET /api/v1/nearby?lat=&lng= — cidades próximas por coordenada (geolocalização)
node-cron scheduler JobLog no DB GET /cities GET /cities/:id GET /fires GET /ranking GET /search GET /nearby
Fase 3
Frontend Core — MVP Visual
4–6 dias Agente C (pode ser subdividido)

Construir as telas principais que definem a identidade visual do produto. Esta fase pode ser dividida entre dois sub-agentes de frontend: um focado no mapa e outro nas páginas de conteúdo.

Agente C1 — Frontend (Mapa)
Mapa Interativo

  • Criar MapPage com mapa do Brasil centralizado
  • Marcadores coloridos por nível de AQI (escala 0–500): verde → marrom
  • Camada de focos de queimada INPE (pontos laranja/vermelho sobrepostos)
  • Clique num marcador → abre tooltip com AQI, cidade, estado e link para dashboard
  • Toggle de camadas: apenas AQI | apenas queimadas | ambos
  • Botão de geolocalização do usuário com marcador de posição
Agente C2 — Frontend (Dashboard)
Páginas de Conteúdo
  • Criar CityPage: gauge de AQI, poluentes em cards, histórico com Recharts, alerta de saúde dinâmico, fonte dos dados
  • Criar RankingPage: lista de cidades mais e menos poluídas, filtros por região e estado
  • Componente SearchBar com autocomplete (debounce 300ms) integrado ao GET /search
  • Componente AqiGauge reutilizável com escala colorida e nível textual
  • Componente HealthAlert: recomendações por faixa de AQI para grupos sensíveis
  • Criar apiService.ts com todos os métodos mapeados para os endpoints da Fase 2
  • Atualizar navegação: Header com links para Mapa, Ranking, Busca
🎨 Design system: reutilizar os tokens de cor e tipografia já definidos na arquitetura (DM Sans, Bebas Neue, DM Mono). O mapa deve usar o mesmo esquema de cor escuro com camadas sobre fundo preto/azul escuro.
MapPage com Leaflet Camada de queimadas CityPage + AqiGauge Recharts histórico SearchBar autocomplete RankingPage HealthAlert component
Fase 4
Fontes Oficiais BR + Features Avançadas
5–7 dias Agentes A + B + C (paralelo)

Integrar as fontes que fazem o diferencial do produto: dados oficiais brasileiros e correlações únicas que nenhuma outra plataforma faz.

Agente B — Data Pipeline
Novas Fontes de Dados
  • CETESBCollector: sistema QUALAR, PM2.5/PM10/O₃/NO₂ hora a hora para SP
  • IEMACollector: 11 estados, dados históricos, 82 localidades
  • IATCollector: IQA diário Região Metropolitana de Curitiba
  • PRODESCollector: alertas de desmatamento INPE (shapefiles → GeoJSON)
  • DATASUSCollector: internações respiratórias por município (API TabNet)
  • IBGECollector: população por município, % idosos, % crianças
  • Atualizar Normalizer para absorver esses novos formatos (XML CETESB, CSV PRODES)
Agente A — Backend
Endpoints Avançados
  • GET /api/v1/cities/:id/health — internações respiratórias + grupos de risco + correlação AQI histórica
  • GET /api/v1/deforestation — áreas desmatadas recentes por estado com correlação queimadas/AQI
  • GET /api/v1/cities/:id/wind-smoke — direção do vento (Open-Meteo) + focos INPE = origem da fumaça
  • GET /api/v1/cities/:id/outdoor-safety — score composto: AQI + UV + pólen + temperatura
  • GET /api/v1/oms-compliance — cidades acima/abaixo do limite PM2.5 OMS (5 µg/m³)
  • Adicionar índice geoespacial PostGIS para queries de proximidade e raio de poluição
Agente C — Frontend
Painéis Avançados
  • Painel de Saúde Pública: gráfico internações × AQI, mapa de calor por município
  • "De onde vem a fumaça?": seta animada de direção do vento + focos INPE destacados
  • Índice "Seguro ao Ar Livre": card verde/amarelo/vermelho com score composto
  • Conformidade OMS: ranking nacional + comparativo global, % de cidades conformes
  • Camada de desmatamento no mapa: polígonos do PRODES com intensidade
🏆 Diferencial único: O cruzamento INPE + Open-Meteo (vento) + DATASUS (internações) é o que nenhuma outra plataforma brasileira faz. Priorizar o endpoint /wind-smoke e o painel de saúde — são o coração do produto.
CETESB + IEMA + IAT PRODES desmatamento DATASUS internações IBGE população GET /health/:city GET /wind-smoke/:city GET /outdoor-safety/:city GET /oms-compliance Painel de saúde pública
Fase 5
Alertas e Notificações
3–4 dias Agentes A + C (paralelo)

Sistema de alertas proativo: usuário define limites e é notificado quando o AQI ultrapassa o threshold em sua cidade favorita.

Agente A — Backend
Sistema de Alertas
  • Schema Prisma: Alert (userId, cityId, threshold, channels: email|push, active)
  • POST /api/v1/alerts — cadastrar alerta (autenticado)
  • GET /api/v1/alerts — listar alertas do usuário
  • DELETE /api/v1/alerts/:id — remover alerta
  • Job AlertCheckerJob: a cada hora verifica AQI das cidades monitoradas e dispara notificações
  • Integrar Resend (ou Nodemailer) para e-mail de alerta
  • Integrar Web Push API (VAPID) para notificações push no browser
Agente C — Frontend
UI de Alertas
  • Tela de configuração de alertas: escolher cidade, definir threshold de AQI, escolher canal
  • Listar alertas ativos com botão de remoção
  • Solicitar permissão de Push Notification no browser
  • Registrar Service Worker para receber push mesmo com app fechado
  • Indicador visual na CityPage se o usuário tem alerta ativo para a cidade
POST/GET/DELETE /alerts AlertCheckerJob E-mail de alerta Web Push (VAPID) UI de gerenciamento de alertas Service Worker

Regras para coordenação de múltiplos agentes
📌 Contrato de Schema
O schema do Prisma é a fonte da verdade. Agente A gera as migrations; Agente B e C nunca alteram o schema diretamente. Qualquer mudança de modelo passa pelo Agente A.
🔀 Branches por fase
Cada agente trabalha em sua branch (feat/phase-N-agente-X). PRs são integrados ao final de cada fase, nunca no meio. Isso evita conflitos durante desenvolvimento simultâneo.
📄 Contrato de API
Agente A define os contratos de response (tipos TypeScript em packages/shared-types/) antes do Agente C construir o frontend. Ambos compartilham o mesmo pacote de tipos via workspace.
🌱 Dado Seed obrigatório
Agente B mantém um script prisma/seed.ts com dados de exemplo (50 cidades, últimas 24h de AQI mockado). Isso permite o Agente C desenvolver o frontend sem depender dos jobs rodando.
🚫 Sem segredos no código
Nenhum agente faz commit de chaves de API, tokens ou senhas. Toda credencial vai para .env local + secrets do GitHub Actions. O Agente D é o guardião dessas regras no CI.
📊 Métricas de sucesso v1
Cobertura de cidades >100, dados atualizando corretamente, mapa carregando em <3s, score Lighthouse >85, zero secrets expostos, CI verde em 100% dos PRs.
